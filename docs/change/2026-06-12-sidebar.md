# Player Sidebar: Queue + Concert Tracks

## Problem

The JS player bar showed a queue badge (count + tooltip) but gave no way to see or manage queued tracks
or browse the other tracks of the currently-playing concert without finding its card on the listing page.
Clicking the track number or title had no effect.

## Design

A `<aside id="player-sidebar">` was added to `#player-container` (outside `#content`, so it survives
htmx navigation). It has two sections stacked vertically:

- **Queue (top)**: client-rendered by `player.js` from the in-memory `queue` array.
- **Concert tracks (bottom)**: server-rendered HTML fetched from
  `GET /concerts/:id/tracks?context=sidebar` and injected via `innerHTML` + `htmx.process()`.

The sidebar slides in from the left using `transform: translateX(-100%)` / `transition`. On screens
wider than 700 px the page content shrinks via `body { margin-left: var(--sidebar-width) }` and the
card grid re-flows naturally (auto-fill). Below 700 px the sidebar overlays the content instead.
Fixed-position elements (`#player-video-panel`) get an explicit `left: var(--sidebar-width)` override
on desktop; on mobile the panel keeps `left: 0` and the sidebar's z-index (1002) places it on top.

The player bar was reordered left-to-right: queue-toggle → track/title info → status → media actions
→ prev/play/pause/next → seek → time.

### Open / close state machine

| State  | Event                                                             | Next   | Side effects |
|--------|-------------------------------------------------------------------|--------|--------------|
| CLOSED | click `#player-queue-toggle` or `#player-track`/`#player-title`  | OPEN   | `body.sidebar-open`; `renderQueue()`; `loadSidebarTracks(state.concertId)` |
| OPEN   | click same controls                                               | CLOSED | remove `body.sidebar-open` |
| OPEN   | `play()` starts a track of a **different** concert                | OPEN   | `loadSidebarTracks(newConcertId)` (detected via `sidebarConcertId`) |
| OPEN   | `play()` starts a track of the **same** concert                   | OPEN   | no re-fetch; `markPlaying` re-highlights all copies |
| OPEN   | htmx card swap of the current concert (`htmx:afterSwap`)          | OPEN   | `loadSidebarTracks` refresh |
| OPEN   | `stopPlayback()`                                                  | CLOSED | remove class; clear both sections; `sidebarConcertId = null` |
| CLOSED | queue mutation (enqueue / dequeue / advance)                      | CLOSED | badge + `renderQueue` updates only |

### Queue entry lifecycle

| State        | Event                                       | Queue effect          | Playback effect        |
|--------------|---------------------------------------------|-----------------------|------------------------|
| not queued   | click track while media playing (file ok)   | append (dupe-guarded) | none                   |
| not queued   | click missing track while playing           | none (prepare flow)   | poll → enqueue on ready |
| queued       | ▶ play-now button (`playQueueEntryNow`)      | splice out            | `startTrack()` now     |
| queued       | ✕ remove button (`dequeue`)                 | splice out            | none                   |
| queued (head)| track ends / Next clicked (`playFromQueue`) | shift                 | play shifted entry     |
| queued       | star toggled (any copy)                     | `entry.liked` flipped | none                   |
| queued       | `stopPlayback()`                            | `queue = []`          | stopped                |

### Sidebar concert-track row

| Row state   | Render                                              | Actions |
|-------------|-----------------------------------------------------|---------|
| available   | star + title btn + [Watch] + trash (JS onclick)     | play/enqueue, like, watch, delete |
| liked       | as above (`:has(.liked)` CSS hides trash)           | play/enqueue, like, watch |
| unavailable | `.track-unavailable` greyed, no trash               | click re-prepares, like |
| playing     | `.playing` bold on ALL copies (sidebar + card)      | click toggles pause |
| preparing   | `.preparing` italic on ALL copies                   | — |

## Key implementation decisions

### Cross-copy sync (Step 1 prerequisite)
The sidebar duplicates every track row. All helpers that assumed a single `[data-concert-id][data-track-idx]`
match were converted to `querySelectorAll`-based functions (`findTrackButtons`, `markPlaying`,
`clearPlaying`, `markPreparing`). Like sync uses the `hx-post` URL as a stable per-track key:
`likeButtonsFor(concertId, trackIdx)` selects every copy's star, and `applyLike()` propagates to
stars, queue entries, and the player bar star in one call.

### `context=sidebar` render mode (Step 2)
`TracksTemplate` gained a `sidebar: bool` flag. Sidebar rows have no `.card` ancestor so the htmx
`hx-target="closest .card"` delete swap cannot work there; the sidebar variant uses
`onclick="Player.sidebarDeleteTrack(id, idx)"` instead. The card path is unchanged.

### Sidebar fetch and race guard
`loadSidebarTracks(concertId)` uses a **generation counter** (`sidebarLoadGen`) checked after each
`await` to discard stale responses when the concert changes rapidly. `sidebarConcertId` is set only on
success; errors clear it to `null` so the next trigger retries the fetch.

### Shared delete core
`postDeleteTrack(concertId, trackIdx)` encapsulates the POST + card-swap + `htmx.process` shared by
the player-bar delete button (`deleteTrack`) and the sidebar trash button (`sidebarDeleteTrack`).
`advanceAfterDelete()` encapsulates the queue-drain → next-track → stop-playback advance logic.
`sidebarDeleteTrack` advances playback only when `state.concertId/trackIdx` matches the deleted track;
deleting a non-playing track from the sidebar never touches audio.

### Queue rendering
`renderQueue()` uses DOM APIs (`textContent`, `createElement`, `append`) throughout — queue track
titles arrive as raw JSON strings and must never be set via innerHTML. Server-rendered sidebar HTML
(trusted, same-origin, Askama-escaped) uses `innerHTML` + `htmx.process()`, matching the htmx
partial-swap pattern used elsewhere in the codebase.

## Trade-offs

- While a prepare chain runs, the card's 3 s status-polling re-swaps the card, which triggers
  `htmx:afterSwap` → `loadSidebarTracks` on each tick. The generation counter keeps correctness;
  scroll position in the sidebar resets but track availability stays fresh.
- `sidebarDeleteTrack`'s disabled-button guard uses an `onclick*=` substring selector. It is correct
  (the closing paren disambiguates overlapping indices) but would be more robust if the template
  passed `this` to the handler.
