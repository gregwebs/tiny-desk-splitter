# Convert frontend JavaScript to strict TypeScript

## Motivation

The three first-party frontend files (`player.js` ~1860 lines, `playlists.js`
~760, `splitter.js` ~680) were hand-written, untyped JS communicating with
the backend via raw `fetch()` and ad hoc JSON shapes. The backend now
publishes an OpenAPI 3.1 spec (utoipa) whose `ToSchema` structs are the exact
wire types these files already consumed by convention — but nothing checked
that the JS actually matched them. A renamed field, an `Option<T>` that
serializes as `null` rather than being omitted, or the `ConcertPlaybackResponse`
tagged union's `mode` discriminator could all silently drift out of sync with
no compile-time signal. Converting to TypeScript compiled under strict
options, with request/response types generated from the live OpenAPI spec,
closes that gap while preserving the existing fast JS dev loop.

## What changed

### Build pipeline

A new `concert-tracker/frontend/` directory holds the TypeScript sources and
build tooling, separate from the Rust crate:

```
concert-tracker/
  frontend/
    package.json, tsconfig.json, build.mjs
    src/
      generated/openapi.d.ts   # generated by openapi-typescript, committed
      api/client.ts            # typed fetch wrappers over the generated types
      shared/                  # dom.ts, globals.d.ts, player-api.ts, playlists-api.ts
      player.ts                 -> static/player.js
      playlists.ts               -> static/playlists.js
      splitter/{core.ts,index.ts} -> static/splitter.js
  static/
    player.js  playlists.js  splitter.js   # @generated — do not edit
    htmx.min.js  style.css                 # untouched
```

`frontend/build.mjs` runs esbuild, bundling each entry point into a
standalone IIFE with the **same filename** the Rust binary already expects
(`concert-tracker/src/web/handlers.rs`'s `player_js`/`playlists_js`/`splitter_js`
embed these via `include_str!`; `static_js_router` serves them from disk in
`--dev`). Output is unminified, has no sourcemap, and carries a `// @generated`
banner — the committed `.js` stays reviewable and reproducible. esbuild's IIFE
format has no shared chunk across entries, so the three modules still
communicate the same way the original hand-written files did: through
`window.Player` / `window.Playlists` / `window.htmx` globals, never via
cross-entry imports.

`concert-tracker/static/*.js` are **committed build artifacts**, not
hand-edited source — `cargo build` stays Node-free. `just ts-verify` rebuilds
from `frontend/src` and fails (`git diff --exit-code`) if the committed `.js`
doesn't match; it's wired into `just lint` and the pre-push hook (blocking).

### OpenAPI-generated types

A new `concert-tracker/src/bin/openapi_dump.rs` binary prints the JSON API's
OpenAPI spec to stdout via the now-public `web::built_api_doc()` (previously
`#[cfg(test)] pub(crate)`) — the same function the existing `openapi.rs` tests
already used to inspect the doc with paths included. `just openapi-types` runs
it and feeds the output to `openapi-typescript`, regenerating
`frontend/src/generated/openapi.d.ts` (committed). `frontend/src/api/client.ts`
re-exports the schema types under their original Rust names (`MediaInfo`,
`PrepareStatus`, `ConcertPlaybackResponse`, the playlists family, etc.) and
wraps `fetch()` in small typed helpers (`getJson`, `getJsonOrNull`, `sendJson`,
`fetchRaw`, `fetchText`) plus named per-endpoint wrappers.

A new integration test, `served_openapi_spec_matches_built_api_doc`
(`concert-tracker/tests/web_integration.rs`), asserts the JSON served at
`/api-docs/openapi.json` equals `built_api_doc().to_json()` exactly — the same
function `openapi-dump` calls — so the two can't silently diverge.

Two shapes worth knowing when reading `client.ts`:
- `Option<T>` fields serialize as `T | null` (the field is always present,
  never omitted, since none of the structs use `skip_serializing_if`). The
  generated types model this as `T | null | undefined` — a deliberately
  looser superset; code treats `null`/`undefined` as equivalent rather than
  relying on a field being absent.
- `ConcertPlaybackResponse` (`#[serde(tag = "mode", rename_all = "snake_case")]`)
  generates as a true TypeScript discriminated union; `api/client.ts` exports
  an `isSourcePlayback()` type guard that narrows it.

### Strict tsconfig

`frontend/tsconfig.json` sets `strict`, `noUncheckedIndexedAccess`,
`exactOptionalPropertyTypes`, `noImplicitReturns`, `noFallthroughCasesInSwitch`,
`noUnusedLocals`/`noUnusedParameters`, and `useUnknownInCatchVariables`, on top
of `target ES2020` / `lib DOM`. `js-tests/tsconfig.json` extends it for the
unit tests. A small `shared/dom.ts` (`byId`/`byIdOrNull`) centralizes the
~50 nullable `getElementById` call sites instead of repeating ad hoc guards.

### Unit tests

`js-tests/splitter.test.js` (CommonJS, `require()`d the compiled bundle's
`_pure` export) is now `js-tests/splitter.test.ts`, importing the pure editor
logic directly from `frontend/src/splitter/core.ts` — same assertions, run via
`node --import tsx --test js-tests/*.test.ts`. The `_pure` export is gone from
the compiled bundle; nothing else referenced it.

### Dev loop

`just dev` now also runs `node frontend/build.mjs --watch` alongside
`cargo-watch` (a shebang recipe with `trap 'kill 0' EXIT` so both processes
exit together). One correction from the prior hot-reload doc
(`2026-06-18-hot-reload-dev-server.md`): editing `static/*.js` was never
actually free of recompilation — `include_str!` embeds those files at compile
time in *both* the dev and prod code paths (only the *serving* path differs at
runtime), so touching them has always invalidated the `handlers.rs`
compilation unit. Measured: ~4s incremental, not a clean rebuild — fast enough
that it wasn't worth adding `notify`/`Reloader` machinery to avoid it.

## Verification

- `just ts-check` clean (strict `tsc --noEmit`, frontend + js-tests).
- `cargo build` / `cargo test -p concert-tracker` — all 73 tests pass,
  including the new `served_openapi_spec_matches_built_api_doc` and the
  updated `prod_router_serves_embedded_js_without_livereload` (now asserts the
  `@generated` banner instead of a literal string from the old hand-written
  JS).
- `npm run test:unit` (12 splitter `core.ts` tests) pass.
- `just lint` clean (`fmt-check` + `clippy` + `ts-check` + `ts-verify`).
- Manual: started `concert-web` against a scratch copy of the e2e fixture DB
  and curl-verified every endpoint each module depends on
  (`/concerts/:id/{media-info,tracks/:idx/media-info,next-media-info,
  concert-playback,split-timestamps,prepare-status}`, the `/api/playlists*`
  family, like/delete) against the generated types, and confirmed the
  rendered templates' inline `onclick="Player.*"/"Playlists.*"/"Splitter.*"`
  call sites match each module's exported API exactly.
- `just dev` smoke-tested: both watchers start (esbuild + cargo-watch) and
  both exit cleanly on Ctrl-C/process termination.
- Playwright e2e: **not run** — chromium fails to launch in this sandbox
  (`browserType.launch: Target page, context or browser has been closed`),
  reproduced identically on unmodified `main`, so it's a pre-existing
  environment limitation, not a regression. Run `npx playwright test` in a
  working environment before merging to get full browser-level coverage.
